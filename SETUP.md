# Real-Time Dynamic Pricing Engine for E-Commerce — Setup Guide

## 1. Overview

This guide explains how to set up the project locally, bring up the Dockerized infrastructure, configure Oracle client access for Go, and run the currently implemented backend and frontend services for testing.

This setup is written for **macOS Apple Silicon (M1/M2/M3)**, but most steps are similar on other systems.

---

## 5. Docker Compose Configuration

Your `docker-compose.yml` should include services for Kafka, Zookeeper, Redis, and Oracle DB.

For Oracle, the recommended setup is to:

* pin the Oracle image version
* auto-create the application user from Compose
* mount SQL/init scripts so the schema is created automatically on first DB initialization

Example Oracle service block:

```yaml
oracle-db:
  image: gvenzl/oracle-free:23.26.1-slim
  container_name: oracle-db
  ports:
    - "1521:1521"
    - "5500:5500"
  environment:
    ORACLE_PASSWORD: oracle
    APP_USER: pricing
    APP_USER_PASSWORD: pricingpass
  volumes:
    - oracle-data:/opt/oracle/oradata
    - ./oracle-db-scripts:/container-entrypoint-initdb.d
```

At minimum, the volume section should include:

```yaml
volumes:
  oracle-data:
```

### Oracle Init Script Folder

The project should contain an `oracle-db-scripts` folder in the repo root.

Recommended contents:

* `01_create_tables.sql`
* `02_run_as_pricing.sh`

Example `01_create_tables.sql`:

```sql
CREATE TABLE price_history (
  id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  product_id VARCHAR2(50),
  price NUMBER,
  event_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE price_overrides (
  id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  product_id VARCHAR2(50) UNIQUE,
  override_price NUMBER,
  reason VARCHAR2(100),
  set_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

Example `02_run_as_pricing.sh`:

```bash
#!/bin/bash
set -e

sqlplus -s pricing/pricingpass@//localhost:1521/FREEPDB1 @/container-entrypoint-initdb.d/01_create_tables.sql
```

Make the shell script executable:

```bash
chmod +x oracle-db-scripts/02_run_as_pricing.sh
```

### Important Behavior

Scripts in `/container-entrypoint-initdb.d` run **only on first database initialization**.
If the Oracle volume already exists, those scripts will **not** run again automatically.

So after changing Oracle init scripts or switching to automated setup for the first time, do a clean reset:

```bash
docker compose down -v
docker compose up -d
```

Start all services:

```bash
docker compose up -d
```

Check running containers:

```bash
docker ps
```

Check Oracle logs:

```bash
docker logs oracle-db
```

You want to eventually see that the database is ready.

Stop services:

```bash
docker compose down
```

Stop services and remove volumes only if you want a clean reset:

```bash
docker compose down -v
```

## 8. Oracle DB Setup

Oracle user creation and schema initialization are expected to happen automatically through Docker Compose and the mounted `oracle-db-scripts` folder.

### Expected Automated Behavior

On first Oracle database initialization:

* `ORACLE_PASSWORD` sets the `SYS` / `SYSTEM` password
* `APP_USER=pricing` creates the application user
* `APP_USER_PASSWORD=pricingpass` sets the application user password
* scripts mounted into `/container-entrypoint-initdb.d` create application tables

### What You Usually Need to Do

Normally, after `docker compose up -d`, you should **not** need to manually create:

* the `pricing` user
* `price_history`
* `price_overrides`

### How to Validate That Automation Worked

Connect as the app user:

```bash
docker exec -it oracle-db sqlplus pricing/pricingpass@//localhost:1521/FREEPDB1
```

Check tables:

```sql
SELECT table_name FROM user_tables;
```

You should see at least:

* `PRICE_HISTORY`
* `PRICE_OVERRIDES`

### If Automation Did Not Run

This usually means one of the following:

* the Oracle volume already existed from an older setup
* the init scripts were added after the first DB initialization
* the mounted script file names or permissions are incorrect

In that case, perform a clean reset so init scripts run again:

```bash
docker compose down -v
docker compose up -d
```

### Manual Fallback (Only If Needed)

If for some reason automatic initialization fails, you can still create the user and tables manually.

Connect as system user:

```bash
docker exec -it oracle-db sqlplus system/oracle@//localhost:1521/FREEPDB1
```

Run:

```sql
CREATE USER pricing IDENTIFIED BY pricingpass;
GRANT CONNECT, RESOURCE TO pricing;
ALTER USER pricing QUOTA UNLIMITED ON USERS;
```

Reconnect as application user:

```bash
docker exec -it oracle-db sqlplus pricing/pricingpass@//localhost:1521/FREEPDB1
```

Create tables manually:

```sql
CREATE TABLE price_history (
  id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  product_id VARCHAR2(50),
  price NUMBER,
  event_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE price_overrides (
  id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY PRIMARY KEY,
  product_id VARCHAR2(50) UNIQUE,
  override_price NUMBER,
  reason VARCHAR2(100),
  set_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 10. Oracle Instant Client for Go (`godror`)

To let Go connect to Oracle on macOS ARM64, install Oracle Instant Client.

Download the macOS ARM64 Instant Client from Oracle’s official site.

Place it somewhere like:

```bash
~/oracle/instantclient
```

Make sure `libclntsh.dylib` exists there.

Example check:

```bash
ls ~/oracle/instantclient/libclntsh.dylib
```

### Important

Oracle connectivity is environment-driven:

* `ORACLE_DSN` default: `pricing/pricingpass@localhost:1521/FREEPDB1`
* `ORACLE_LIB_DIR` optional: only set when Oracle Instant Client path must be forced
* `ORACLE_TIMEZONE` default: `UTC` (recommended to avoid `godror` timezone warnings)

If `ORACLE_LIB_DIR` is not set, apps use `ORACLE_DSN` as-is.

## 12. Runtime Configuration (Phase 5)

All backend services use shared env-based config with local-first defaults:

* `KAFKA_BROKERS=localhost:9092`
* `REDIS_ADDR=localhost:6379`
* `PRODUCT_EVENTS_TOPIC=product-events`
* `PRICE_UPDATES_TOPIC=price-updates`
* `API_PORT=8080`
* `ADMIN_API_KEY=local-admin-key` (**required** for API server startup)
* `API_KEY_HEADER=X-API-Key` (optional, default shown)
* `ORACLE_DSN=pricing/pricingpass@localhost:1521/FREEPDB1`
* `ORACLE_LIB_DIR=` (optional)
* `ORACLE_TIMEZONE=UTC`

The Go services auto-load values from a `.env` file in the repo root, so you can set local values once there and run commands without prefixing env vars each time.

Frontend proxy routes use:

* `BACKEND_BASE_URL=http://localhost:8080` (default)
* `BACKEND_API_KEY_HEADER=X-API-Key` (default; should match backend `API_KEY_HEADER`)
* `NEXT_PUBLIC_PRICE_WS_URL=ws://localhost:8080/ws/prices` (default WebSocket stream URL)

You can keep this in `frontend/.env.local` (optional):

```bash
BACKEND_BASE_URL=http://localhost:8080
BACKEND_API_KEY_HEADER=X-API-Key
NEXT_PUBLIC_PRICE_WS_URL=ws://localhost:8080/ws/prices
```

### Override Examples

Custom API port + Redis:

```bash
API_PORT=8090 REDIS_ADDR=localhost:6380 go run ./backend/cmd/api-server
```

Custom Oracle DSN + lib dir:


## 14. What Each Service Does

### Event Producer

* Simulates fake user activity
* Sends `view`, `cart`, and `purchase` events to Kafka topic `product-events`

### Pricing Engine

* Consumes Kafka events
* Seeds baseline catalog state at startup (`demand=0`, base Redis price, and ensures a one-time `base` row in Oracle history)
* Updates demand score in Redis using weighted values
* Computes dynamic prices
* Stores latest price in Redis for each valid consumed event
* Inserts price history into Oracle DB for each valid consumed event
* Publishes latest price to Kafka topic `price-updates` with bounded retry backoff

### API Server

* Exposes:
  * `GET /ws/prices` (WebSocket live stream)
  * `GET /price/{productID}`
  * `POST /override`
  * `DELETE /override/{productID}`
  * `GET /history/{productID}?limit=N`
* Protects `POST /override`, `DELETE /override/{productID}`, and `GET /history/{productID}` with static API key auth
  * Header name: `X-API-Key` by default (or configured `API_KEY_HEADER`)
  * Missing/invalid key returns `401 {"error":"unauthorized"}`
* `GET /price` returns `product_id`, `price`, `demand`, and `price_source`
* Applies Oracle override price when present, otherwise returns Redis live price

### Frontend Dashboard (Next.js)

* Single-page dashboard for static catalog `PA` to `PJ`
* Uses WebSocket stream from API server for live price updates (no timer polling loop)
* Uses Next.js proxy routes:
  * `GET /api/price/[productId]`
  * `POST /api/override`
  * `DELETE /api/override/[productId]`
  * `GET /api/history/[productId]`
* Supports inline override set/clear and price-history chart

## 15. Test the API and Dashboard

Example price request:

```bash
curl http://localhost:8080/price/PA
```

Expected response format:

```json
{
  "product_id": "PA",
  "price": 123.45,
  "demand": 7,
  "price_source": "dynamic"
}
```

Create/update override:

```bash
curl -X POST http://localhost:8080/override \
  -H 'X-API-Key: local-admin-key' \
  -H 'Content-Type: application/json' \
  -d '{"product_id":"PA","override_price":149.99,"reason":"campaign override"}'
```

Note: override creation now requires the product to already exist in live Redis state (demand/price keys). If the product has not been seen yet, the API returns `404 product not found`.

Delete override:

```bash
curl -X DELETE http://localhost:8080/override/PA \
  -H 'X-API-Key: local-admin-key'
```

Fetch product history:

```bash
curl http://localhost:8080/history/PA?limit=20 \
  -H 'X-API-Key: local-admin-key'
```

History row shape includes `price_source` (`base`, `dynamic`, `override`) so charts can distinguish baseline/event/override points.

If you get `Price not found`, make sure:

* Docker services are up
* pricing engine is running
* event producer has generated events for that product

Quick UI validation flow (Phase 5):

1. Open `http://localhost:3000`.
2. Enter a valid Operator API key (`local-admin-key` if using the sample `.env`).
3. Confirm rows `PA` to `PJ` appear and update in real time while events are produced.
4. Select a product row and verify the history chart loads.
5. Set an override from that row and verify `price_source` changes to `override`.
6. Remove the override and verify `price_source` returns to `dynamic`.

Optional WebSocket integration smoke test (requires Docker services + pricing-engine + api-server running):

```bash
go test -tags=integration ./backend/integration -run TestWebSocketPriceUpdateSmoke -count=1
```

## 17. Useful Troubleshooting

### Kafka topics not found

Re-run:

```bash
go run ./tools/create_topics.go
```

### Redis appears empty

Check whether the pricing engine is running and receiving events.

### Oracle container fails on Apple Silicon

Use a pinned Oracle Free image such as:

* `gvenzl/oracle-free:23.26.1-slim`

If Oracle starts but the expected app user or tables are missing, the init scripts probably did not rerun because the Oracle volume already existed. In that case:

```bash
docker compose down -v
docker compose up -d
```

### `godror` cannot find Oracle client library

If you set `ORACLE_LIB_DIR`, make sure Oracle Instant Client is downloaded and the path is correct.

### API returns wrong price

If an override exists in Oracle, `GET /price` intentionally returns override price. Delete the override to return to dynamic Redis pricing.

### Need to stop a long-running Go service

In the terminal running it:

```bash
Ctrl+C
```

## 18. Current Implemented End-to-End Flow

1. Docker starts Kafka, Redis, and Oracle
2. Kafka topics are created
3. Event producer sends fake product events
4. Pricing engine consumes them and computes updated prices
5. Redis stores current demand and current price
6. Oracle stores price history
7. API server returns live price and demand (override-aware)
8. Next.js dashboard receives live updates through `/ws/prices` and invokes override/history APIs through proxy routes
